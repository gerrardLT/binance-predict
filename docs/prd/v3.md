# BTC 5min LLM 预测系统需求文档 V3

> 版本说明：V3（本次修订）相对 V2 的核心变化
>
> 1. **「情绪曲线自进化 Agent Loop」成为系统唯一的自动决策引擎**。该引擎不依赖人工预定义的形态库，而是让 LLM 在四阶段闭环（学习→预测→验证→进化）中自主发现、命名、固化、淘汰情绪曲线模式，由 tracker/archiver 事件全自动驱动运转。
> 2. **彻底清除路径 A（K 线 + 决策 LLM，V1/V2 遗留）及其全部外围设施**：删除 `decision_job`/`verify_job`/`kline_event_listener`/心跳/准确率任务/SSE 广播、27 个路径 A API 端点、7 张路径 A 数据表、`decide()`/`review()` 两个 LLM 调用点、`feature_service.py`/`rule_service.py`/`prompts/templates.py` 三个模块，以及 `_auto_predict`/`_auto_trade`/`scheduler` 等全局状态与 APScheduler 依赖。路径 A 不再以任何形式存在。
> 3. **保留概率动量算法（路径 C，`MomentumService`）但不参与自动运转**：纯算法、不依赖 LLM、不依赖 K 线，仅作为手动可调用的备选分析方案，不落库、不下单。
> 4. **前端重构为两个页面**：`frontend/src/App.tsx` 删除旧四个 tab，改为「市场情绪」+「Agent 自进化」两页面，接入 `/api/sentiment/agent/*` 系列端点。
> 5. **交易门控简化为路径 B 单一入口，且无任何总开关**：随路径 A 一并移除的 `_auto_trade` 开关消失后，当前**没有任何开关能阻止 Agent Loop 自动下单**（重要风险点，见第 9 节），需要用户明确知晓。
> 6. **三张 Agent 数据表**（`pattern_memory`、`agent_predictions`、`pattern_change_log`）延续「LLM 输出 = 数据库列 = API 响应」单一事实源原则。

## 1. 项目定位（更新）

系统已从"由 LLM 直接主导单次判断"演变为"由自进化 Agent 主导的持续学习判断系统"。核心差异：

```text
V2：每次决策都是独立的一次性 LLM 调用，判断逻辑由固定 prompt + 用户规则决定。
V3：情绪曲线 Agent 通过四阶段闭环持续从真实市场结果中学习、命名、固化、淘汰"模式"（Pattern），
    判断逻辑随运行时间自主进化，不依赖人工预定义的形态清单。
```

V1 的机器学习模型路径、V2 的"移除 ML，纯 LLM 判断"路径（统称路径 A：K 线 + 决策 LLM），在本次修订中已被**彻底删除**——相关任务、端点、数据表、LLM 调用点与模块文件全部移除，不再作为遗留路径保留。系统现在只有情绪曲线自进化 Agent 一个自动运转的判断引擎。

## 2. 系统架构（单一自动引擎 + 手动备选）

路径 A 清除后，系统只剩一条自动运转的路径（B）与一条手动备选路径（C），两者共享同一份预测市场采样数据，但彼此独立。

### 2.1 路径总览

```text
路径 B：情绪曲线自进化 Agent Loop（SentimentAgent + AgentScheduler）★ 系统唯一的自动决策引擎
  触发：全自动，由 tracker/archiver 事件驱动（见第 3 节）
  数据：sentiment_windows / pattern_memory / agent_predictions / pattern_change_log
  验证：Validate 阶段自动进行，按时间范围匹配预测与窗口结果
  交易：SentimentAgent 内部门控 should_trade（confidence 严格 > 0.6 且方向 UP/DOWN 即真实下单），
        无任何总开关可阻止（见第 9 节风险提示）

路径 C：概率动量算法（MomentumService）★ 手动备选，不生效于自动流程
  触发：仅 POST /api/sentiment/momentum-predict 手动触发
  数据：不落库，基于内存 _pm_history 实时计算，纯算法不调用 LLM
  验证：无自动验证
  交易：无交易执行，仅返回分析结果供参考
```

### 2.2 数据流与路径关系图

```text
Binance Prediction Markets（只读报价） ──→ _pm_history（15s 采样）──→ sentiment_windows（5min 归档）
                                                    │                        │
                                                    │ 第 10 采样点            │ 归档完成
                                                    ▼                        ▼
                                          [路径 B] PREDICT 事件      [路径 B] WINDOW_ARCHIVED 事件
                                                    │                        │
                                                    ▼                        ▼
                                          SentimentAgent.predict   Validate → Learn
                                          (冷启动检查 ACTIVE=0?)    (回填结果 → 模式统计)
                                                    │                        │
                                                    ▼                        ▼
                                          agent_predictions 表      pattern_memory 表
                                          + 交易门控(阈值0.6,无总开关)  + pattern_change_log 表
                                                                             │
                                                            累计12次Validate ▼
                                                                      Evolve（自我反思）

_pm_history（同一份数据） ──→ [路径 C] MomentumService.analyze()（手动触发，纯算法，独立于路径 B）

注：原路径 A（K 线 + 决策 LLM）及其数据管道（MarketDataStore→FeatureService→decision_job→
    predictions/verify_job）已整体删除，不再出现在数据流中。
```

## 3. 情绪曲线自进化 Agent Loop（V3 核心新增）

### 3.1 设计理念

不同于路径 A/V2 依赖人工在 prompt 中预定义 market_state 枚举与判断层级，Agent Loop 的核心理念是：**不预定义任何情绪曲线形态，完全由 LLM 在运行中自主发现、命名、验证、淘汰模式**。程序（Harness）只负责数据管道、定时触发和存储，不干预 LLM 的认知内容。

### 3.2 四阶段闭环

```text
学习 Learn    → 窗口归档后，分析最近 50 个历史窗口 + 当前 ACTIVE 模式 → LLM 发现新模式或更新旧模式
预测 Predict  → 窗口进行中第 10 个采样点（约150秒）→ LLM 将当前曲线与 ACTIVE 模式匹配 → 输出方向+置信度
验证 Validate → 窗口归档后 → 按时间范围匹配该窗口内的预测记录 → 回填 is_correct，更新模式 win_rate
进化 Evolve   → 累计完成 12 次 Validate 后 → LLM 自我反思全部模式表现 → RETAIN/MODIFY/RETIRE/CREATE
```

### 3.3 触发条件与调度（真实实现，均可通过 .env 覆盖）

```text
Predict 触发：当前窗口累计采样点 >= agent_predict_trigger_samples（默认 10，约 150 秒）时触发，
              同一窗口只触发一次
Validate 触发：情绪窗口归档完成（WINDOW_ARCHIVED 事件）后触发，按 prediction_time 落在窗口
              [start_time, end_time] 范围内匹配该窗口的未验证预测
Learn 触发：紧随 Validate 之后触发（同一 WINDOW_ARCHIVED 事件），或系统启动时 ACTIVE 模式数为 0
           时立即触发一次（冷启动）
Evolve 触发：累计完成 agent_evolve_interval（默认 12）次 Validate 后触发

调度方式：AgentScheduler 用 asyncio.PriorityQueue 单 worker 串行消费，保证同一时刻至多一个阶段
         执行；优先级 PREDICT(0) > WINDOW_ARCHIVED(1) > EVOLVE(3)，Predict 时间敏感，永远优先出队

分阶段超时（外层硬兜底，均可 .env 覆盖）：
  Validate = 10s（无 LLM 调用）
  Predict  = 30s（内层 LLM 超时 25s）
  Learn    = 110s（内层 LLM 超时 100s）
  Evolve   = 110s（内层 LLM 超时 100s）
超时/LLM 调用失败均不静默降级：Predict 落库 NO_TRADE 并记录失败原因；Learn/Evolve 跳过本次且模式库
不变，记录完整错误日志。
```

### 3.4 冷启动与模式数量约束

```text
冷启动（ACTIVE 模式数 = 0）：
  - Predict 直接输出 NO_TRADE，reasoning 标注"模式库为空，等待学习积累"，不调用 LLM
  - AgentScheduler 启动时立即直调一次 Learn（不经事件队列），用最近 50 个历史窗口做初始模式发现

模式数量上限（agent_active_pattern_cap，默认 30）：
  - Learn 阶段：软约束，prompt 引导 LLM 接近上限时优先 UPDATE 已有模式而非新建
  - Evolve 阶段：硬约束，ACTIVE 超过上限时，在 sample_count > agent_min_sample（默认 5）的模式中
    按 win_rate 升序强制淘汰超额部分；样本不足的新模式受保护，不会被上限淘汰
  - 冷启动保护：ACTIVE 模式数 < 3 时，Evolve 忽略所有 RETIRE 操作（含 LLM 建议的与上限触发的）
```

### 3.5 验证判定真值表

```text
预测方向 predicted_direction | 实际结果 actual_outcome | is_correct
UP                            | UP                       | True（方向命中）
DOWN                          | DOWN                     | True（方向命中）
UP                            | DOWN                     | False（方向相反）
DOWN                          | UP                       | False（方向相反）
UP 或 DOWN                    | NOISE                    | False（有方向判断但市场无显著波动）
NO_TRADE                      | NOISE                    | True（正确规避非行情）
NO_TRADE                      | UP 或 DOWN               | False（错失机会）
```

win_rate 由 Harness 维护的 correct_count / sample_count 精确计算（非 LLM 自报），Harness 维护字段
`correct_count` 不属于 LLM 输出契约。

## 4. LLM 调用点清单（V3：共 3 处，全部属于路径 B Agent Loop）

路径 A 删除后，原 `decide()`/`review()` 两个 LLM 调用点已从 `llm_service.py` 移除。当前系统仅存 3 处 LLM 调用，均由 Agent Loop 自动触发：

```text
调用点              触发路径      触发时机                    模型                  自动重试
agent_learn()       路径B(自动)   Learn阶段                   settings.decision_model  2次
agent_predict()     路径B(自动)   Predict阶段                 settings.decision_model  2次
agent_evolve()      路径B(自动)   Evolve阶段                  settings.decision_model  2次
```

全部 3 处均使用同一套已验证的 Instructor 调用形态：`instructor.from_openai(openai.AsyncOpenAI(...))` +
`Mode.TOOLS`（Tool Calling，因模型不支持原生 Structured Output）+ `client.create(response_model=...,
max_retries=2, extra_body={"thinking":{"type":"disabled"}})`。Pydantic 校验失败时 Instructor 自动将
错误回传 LLM 修正，最多重试 2 次；仍失败则抛出异常，由上层按"无静默降级"策略处置（Predict 判 NO_TRADE
并记录失败原因、Learn/Evolve 跳过本次不写入部分数据）。

三处调用均使用 `settings.decision_model`（默认 "deepseek-v4-pro"，可经 .env 覆盖，代码零硬编码）。
注意：`settings.review_model`（默认 "qwen3.7-max"）随 `review()` 删除后**已无任何使用者**，仅作为历史
配置项保留（见第 10 节），建议后续通过 context7/官方文档核验模型 ID 时一并决定是否清理。

```text
枚举：PatternStatus(ACTIVE|RETIRED|EVOLVING)、PatternDirection(UP|DOWN)、
```

## 8. 提示词与自主命名原则（延续 V2 reasoning-first，新增自主命名约束）

```text
Agent Loop 的三套 system prompt（Learn/Predict/Evolve）均遵循（路径 A 的 decide/review 提示词已随代码删除）：
1. reasoning-first：Pydantic Schema 中 reasoning 字段置于结论字段之前，强制先输出推理过程
2. Agent Loop 三套提示词额外要求：不依赖任何预定义模式名称或形态清单，LLM 必须依据观察到的
   曲线特征自行命名与描述模式（如"背离衰竭形态""同向加速形态"等，具体名称完全由 LLM 决定）
3. 证据驱动：新建/更新/淘汰模式都需要有可观察的曲线特征支撑与足够历史样本印证
```

## 9. 重要风险提示：交易门控无总开关（必须明确告知用户）

路径 A 删除时，其 `_auto_trade` 开关（及 `GET/PUT /api/settings/auto-trade` 端点）一并移除。系统现在只剩路径 B 一套交易门控，且**没有任何总开关能在下单前统一拦截**。

```text
当前唯一的交易门控（SentimentAgent._write_prediction_and_trade 内，纯函数 should_trade）：
  仅当 predicted_direction ∈ {UP, DOWN} 且 confidence 严格 > agent_trade_confidence_threshold
  （默认 0.6）时，即无条件调用 BinancePredictionTrader.execute_trade 执行真实下单；
  其余情形跳过并写非空 skip_trade_reason（记录而非静默）。
  该判定不依赖任何用户可切换的开关，唯一可调的是 .env 中的置信度阈值本身。

⚠ 实际影响：只要 Agent Loop 在 Predict 阶段命中某个 ACTIVE 模式且置信度超过 0.6，系统就会用
  真金白银下单（单笔金额取 .env 的 PREDICTION_TRADE_AMOUNT_USDT，当前为 2 USDT），无任何开关
  可临时叫停。这是当前代码的真实行为，不是本文档的假设。

建议（实盘前必做）：补充一个 agent_auto_trade 安全开关（.env 或运行时可切换），在 should_trade
  通过后、execute_trade 之前做最终拦截，使运营方能一键停用真实下单而不影响预测与学习。此项应作为
  后续需求单独提出并实现。
```

## 11. 验收标准（新增，针对 Agent Loop）

```text
Agent Loop 验收：
  Predict 阶段冷启动（ACTIVE=0）时输出 NO_TRADE 且不调用 LLM
  Validate 阶段能通过时间范围正确匹配 Predict 时尚未归档窗口（sentiment_window_id 为空）产生的预测
  每次模式变更（CREATE/UPDATE/RETIRE）恰好生成一条 pattern_change_log，且在同一事务提交
  ACTIVE 模式数超过 30 时，Evolve 仅淘汰 sample_count>5 的模式中 win_rate 最低者
  ACTIVE 模式数少于 3 时，Evolve 不执行任何淘汰（含 LLM 建议的）
  单 worker 调度下同一时刻至多一个阶段执行；PREDICT 事件相对 WINDOW_ARCHIVED/EVOLVE 优先出队
  LLM 调用失败/超时时 Predict 落库 NO_TRADE 并记录失败原因；Learn/Evolve 跳过且模式库不变
一致性验收（延续V2，新增3张表）：
  pattern_memory / agent_predictions / pattern_change_log 的 LLM 输出Schema = DB列 = API响应
  三者字段与枚举完全一致
```
