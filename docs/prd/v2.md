# BTC 5min LLM 预测系统需求文档 V2

> 版本说明：V2 相对 V1 的核心变化
> 1. **移除全部机器学习模型（LightGBM/XGBoost 等）与回测/walk-forward 环节**。最终判断完全由 LLM 完成。
> 2. **新增「用户自定义规则注入」为一等功能**：用户编写的规则会在运行时注入到 LLM 的 prompt 中，直接影响判断。
> 3. **统一数据契约**：LLM 输出结构 = 数据库列 = API 响应，三者由同一份 JSON Schema 派生，枚举与字段名全程一致。
> 4. Agent 1-4（行情/盘口/成交流/合约情绪）退化为**确定性特征计算**，不再是独立 LLM 调用；系统只保留两处 LLM 调用：**决策 LLM** 与 **复盘 LLM**。

---

## 1. 项目名称

**BTC 5min LLM 全自动预测系统**

---

## 2. 项目定位

构建一个由 LLM 直接主导判断的 BTC 短周期预测系统。

系统持续采集市场数据，将原始数据加工成结构化的「市场快照（feature snapshot）」，连同**用户自定义规则**一起注入到 LLM 的 prompt 中，由 LLM 像一个「自动化交易分析员」一样，判断未来 5 分钟 BTCUSDT 价格方向，并输出结构化预测结果。

第一阶段只做：

```text
BTC/USDT
5分钟方向预测
LLM 自动判断
用户自定义规则注入
预测结果记录
预测复盘（LLM 归因）
规则/提示词迭代（人工确认后生效）
```

暂不自动下单。

与 V1 的根本区别：**判断来自 LLM 的推理，而非机器学习模型的概率输出**。系统不追求统计意义上的胜率证明，而是让 LLM 基于实时数据 + 人类经验规则做出可解释的判断。

---

## 3. 核心目标

```text
自动读取市场数据
自动加工成结构化市场快照
将用户规则注入 LLM prompt
LLM 判断 BTC 未来 5 分钟方向（UP / DOWN / NO_TRADE）
LLM 输出上涨概率、下跌概率、置信度、理由、反向风险
自动记录每一次预测
5 分钟后自动验证预测结果
LLM 自动归因错误
LLM 产出规则/提示词优化建议（人工确认后生效）
```

系统输出为完整结构化结论（见第 7 节数据契约）。

---

## 4. 一句话定义

```text
一个由 LLM 主导、支持人类规则注入的 BTC 5分钟市场判断系统
```

LLM 的职责：看数据 → 结合规则理解市场 → 做判断 → 给概率 → 给理由 → 做复盘 → 提出规则修正建议。

## 6. 系统架构（纯 LLM）

### 6.1 整体流水线

```text
数据采集（Binance 现货 + 合约，真实接口）
  ↓
特征计算（确定性计算，无 LLM）→ 结构化市场快照 feature snapshot
  ↓
规则注入（系统基础规则 + 用户启用的自定义规则，按优先级拼装）
  ↓
决策 LLM（读取 快照 + 规则 → 输出 UP/DOWN/NO_TRADE + 概率 + 理由）
  ↓
输出校验（JSON Schema 校验 + 硬规则代码兜底）→ 预测落库
  ↓
5 分钟后自动验证（真实价格对比）
  ↓
复盘 LLM（归因错误类型 + 产出规则优化建议）
  ↓
经验库沉淀（可选：Vector DB / RAG 回灌历史误判案例）
```

### 6.2 LLM 调用点（仅两处）

| 调用点 | 触发时机 | 输入 | 输出 |
|--------|----------|------|------|
| 决策 LLM | 每 60 秒一次预测周期 | 市场快照 + 注入规则 | 预测 JSON（见 7.1） |
| 复盘 LLM | 每条预测到期后（T+5min） | 预测记录 + 实际结果 + 历史相似案例 | 复盘 JSON（见 7.3） |

Agent 1-4（行情观察员/盘口分析员/成交流分析员/合约情绪分析员）**不再是独立的 LLM 调用**，而是特征计算服务中的确定性计算模块，各自产出结构化子块，合并进市场快照。这样每分钟只有 1 次决策 LLM 调用，控制成本与延迟。

### 6.3 技术选型须联网核验（对应用户规则 4/6）

进入设计前，须用 context7 / 官方文档核验以下项的**最新用法**，不得凭记忆写死：

```text
LLM 结构化输出：OpenAI structured outputs / function calling、Anthropic tool use、Pydantic 模型约束
Binance 现货 REST/WebSocket 市场数据接口与限频权重
Binance USDⓈ-M 合约 REST 接口（资金费率、OI、多空比、taker 买卖量）周期与历史长度限制
官方 connector SDK（binance-connector-python / binance-futures-connector-python）最新版本与维护状态
（可选）Vector DB 选型：pgvector / Milvus / Qdrant
```

## 7. 数据契约（单一事实来源 · Single Source of Truth）

> 本节是全系统的一致性基准。LLM 输出、数据库列、API 响应必须与此保持一致。所有枚举**大小写固定**。

### 7.1 决策 LLM 输出 Schema

```json
{
  "symbol": "BTCUSDT",
  "horizon": "5m",
  "prediction_time": "2026-07-02T10:00:00Z",
  "target_time": "2026-07-02T10:05:00Z",
  "final_prediction": "UP",
  "p_up": 0.64,
  "p_down": 0.36,
  "confidence": 0.72,
  "risk_level": "MEDIUM",
  "market_state": "MOMENTUM_UP",
  "reason": [
    "1m 与 3m 收益率同向为正，短周期动量一致",
    "主动买入比例高于主动卖出比例",
    "盘口买盘深度占优，imbalance_5 达到 0.63",
    "合约持仓量上升，价格同步上涨"
  ],
  "reverse_risk": [
    "若盘口买单快速撤退，可能形成假突破",
    "若 1 分钟内主动卖出量放大，需取消看涨判断"
  ],
  "invalid_conditions": [],
  "applied_rule_ids": [12, 27],
  "rules_version": "rules_v5",
  "prompt_version": "decision_prompt_v3"
}
```

**字段约束**

```text
final_prediction : 枚举 UP | DOWN | NO_TRADE
p_up, p_down     : 0~1 浮点，且 p_up + p_down ≈ 1（NO_TRADE 时可均为 0）
confidence       : 0~1 浮点（LLM 主观置信度，非统计概率，见第 11 节）
risk_level       : 枚举 LOW | MEDIUM | HIGH
market_state     : 固定枚举集合（见 7.4）
reason           : 字符串数组，至少 1 条
reverse_risk     : 字符串数组，NO_TRADE 时可为空
invalid_conditions : 触发的失效/异常条件数组
applied_rule_ids : 本次实际注入并生效的用户规则 id 数组
rules_version    : 本次使用的规则集版本
prompt_version   : 本次使用的决策提示词版本
```

```json
{
  "prediction_id": "20260702_100000_BTCUSDT",
```

## 8. 用户自定义规则注入（V2 核心功能）

### 8.1 目标

允许用户用自然语言编写交易判断规则，这些规则在每次决策时注入到 LLM 的 prompt，直接影响 LLM 的判断。规则可增删改、启用/停用、排优先级，并可追踪每条规则上线后的表现。

### 8.2 规则类型

```text
HARD（硬规则）：必须严格遵守。示例：spread_ratio > 0.0002 一律 NO_TRADE。
                 硬规则除了注入 prompt，还必须在代码侧做一道兜底校验（不能只靠 LLM 自觉）。
SOFT（软偏好）：倾向性引导。示例：倾向于只在放量时给方向。仅通过 prompt 影响 LLM。
```

### 8.3 注入方式

决策 prompt 模板中预留两段占位：

```text
{{system_rules}}   系统基础原则（见第 10 节，不可被用户规则覆盖）
{{user_rules}}     启用中的用户规则，按 priority 升序拼装；同类冲突时 priority 小者优先
```

运行时组装逻辑：

```text
1. 读取所有 enabled = true 的用户规则，按 priority 排序。
2. HARD 规则先于 SOFT 规则注入，并显式标注"这是必须遵守的硬性约束"。
3. 生成本次 rules_version 快照（内容哈希），写入预测记录的 applied_rule_ids 与 rules_version。
4. 决策 LLM 返回后，对 HARD 规则做代码兜底校验；若 LLM 输出违反 HARD 规则，强制改写为 NO_TRADE 并记录 invalid_conditions。
```

### 8.4 冲突优先级

```text
系统基础安全原则（第 10 节）  >  HARD 规则  >  SOFT 规则  >  LLM 自主判断
```

用户规则与系统安全原则冲突时，系统原则优先，并在该次预测的 invalid_conditions 中记录冲突。

### 8.5 规则效果追踪

每条预测记录其 `applied_rule_ids` 与 `rules_version`。复盘统计时可按规则维度聚合：某条规则启用后，命中该规则的预测准确率、NO_TRADE 比例变化等，供用户判断规则是否有效。

## 9. 预测逻辑设计

### 9.1 三种预测结果

```text
UP        ：预测未来 5 分钟上涨
DOWN      ：预测未来 5 分钟下跌
NO_TRADE  ：市场不清晰 / 数据异常 / 规则触发 / 置信度不足，不做方向判断
```

`NO_TRADE` 是高质量判断的一部分，不是失败。

### 9.2 LLM 判断层级（写入 prompt 引导）

```text
第一层 市场是否可预测：行情是否活跃、成交量是否足够、盘口是否正常、spread 是否过大、
                       是否异常跳价、是否有数据延迟/断流。不可预测 → NO_TRADE。
第二层 短周期方向：10s / 30s / 1m / 3m / 5m 方向是否一致。
第三层 资金推动：上涨是否有主动买入推动、下跌是否有主动卖出推动、是否只是低流动性噪声。
第四层 合约确认：现货与合约是否同步、OI 变化、多空比是否极端、资金费率是否过热。
第五层 规则与最终判断：应用用户规则，综合输出 UP/DOWN/NO_TRADE + 概率 + 置信度 + 理由 + 反向风险。
```

### 9.3 预测结果标签定义（用于验证）

```text
预测时间 T，目标时间 T+5min
入场参考价 entry_price = mid_price(T)
验证价格 exit_price = mid_price(T+5min)
future_return = exit_price / entry_price - 1

future_return >  noise_threshold  → actual_label = UP
future_return < -noise_threshold  → actual_label = DOWN
其他                              → actual_label = NOISE

noise_threshold 可配置，默认 0.05%（范围 0.03% ~ 0.08%）
is_correct = (final_prediction == actual_label)；NO_TRADE 不计入方向准确率，单独统计。
```

## 10. 系统基础提示词（决策 LLM）

```text
你是 BTC 5 分钟短周期预测系统的决策 LLM。

任务：根据结构化市场快照与注入的规则，判断 BTCUSDT 未来 5 分钟更可能上涨、下跌，还是不适合预测。

必须遵守（系统安全原则，优先级高于任何用户规则）：
1. 只根据输入数据判断，不得凭空编造行情。
2. 数据矛盾或置信度不足时，优先输出 NO_TRADE。
3. anomaly_flags 中任一异常为 true 时，优先输出 NO_TRADE。
4. 宁可少预测，也不要乱预测。
5. 不能只因价格上涨就判断继续上涨，也不能只因价格下跌就判断继续下跌。
6. 必须识别假突破、诱多、诱空、流动性不足、异常波动。
7. 严格遵守标注为 HARD 的用户规则；参考标注为 SOFT 的用户偏好。
8. 输出必须是严格 JSON，字段与系统数据契约一致，枚举值大小写固定。
9. 只输出预测信号与风险解释，不输出投资建议。

{{system_rules}}
{{user_rules}}

输出字段：final_prediction, p_up, p_down, confidence, risk_level, market_state,
reason, reverse_risk, invalid_conditions, applied_rule_ids, rules_version, prompt_version。
```

## 11. 置信度说明（重要）

V2 的 `confidence` 是 **LLM 的主观置信度**，不是经过统计校准的概率。它反映 LLM 对本次判断的把握程度，用作运营门槛，不应被解读为「历史胜率」。

运营参考门槛（可配置）：

```text
confidence >= 0.75 : 强信号
0.60 <= confidence < 0.75 : 普通信号
0.50 <= confidence < 0.60 : 弱信号
confidence < 0.50 : 视为 NO_TRADE
```

## 12. 复盘与自我优化机制

### 12.1 每条预测到期后复盘

预测到 T+5min 后，系统读取真实价格，计算 actual_label 与 is_correct，交由复盘 LLM 归因 error_type、生成 lesson，并可产出 rule_suggestions。

### 12.2 允许 LLM 产出、但需人工确认才生效的内容

```text
决策提示词修改建议
新增 / 修改 / 停用 用户规则的建议
NO_TRADE 条件调整建议
置信度门槛调整建议
```

> V2 原则：**LLM 只产出建议，不自动改线上判断逻辑**。所有变更由用户在规则/提示词管理界面确认后生效，并生成新版本号。

### 12.3 不允许 LLM 触碰的内容

```text
实盘资金比例、是否开启自动交易、最大亏损限制、
API Key 权限、交易账户权限、杠杆倍数
```

### 12.4 版本管理

```text
提示词版本：decision_prompt_v1, v2, v3 ...
规则集版本：rules_v1, v2, v3 ...（每次规则变更生成内容哈希快照）
每个版本记录：修改原因、修改内容、适用场景、预期改善点、回滚条件。
```

## 13. 数据需求（真实接口，无 fallback、无伪造数据）

### 13.1 现货行情（Binance Spot 公开市场数据，无需 API Key）

```text
实时价格 / bookTicker
1m、5m K线
实时成交 trade / 聚合成交 aggTrade
订单簿 depth
```

### 13.2 合约行情（Binance USDⓈ-M Futures）

```text
永续合约价格、1m/5m K线、标记价格、指数价格
资金费率、持仓量 OI 及变化、多空比、top trader 多空比、主动买卖量
```

> 已知约束：合约多空比等 `/futures/data/*` 接口官方仅提供最近约 30 天数据；周期支持 5m/15m/30m/1h。具体以联网核验为准。

### 13.3 数据质量原则（对应用户规则 3/7）

```text
只使用真实接口返回的数据。
数据缺失 / 延迟 / 异常时，明确置 anomaly_flags 并进入 NO_TRADE，绝不用伪造或默认值填充后当真实数据判断。
备选方案（如 WS 断线降级为 REST 轮询）允许，但必须记录数据来源与质量，不做静默处理。
```

## 14. 特征计算（确定性，无 LLM）

沿用 V1 第 10 节的特征清单（价格 / 成交量 / 盘口 / 主动买卖 / 合约情绪 / 异常过滤），产出结构化市场快照。核心公式：

```text
imbalance = bid_depth / (bid_depth + ask_depth)
```

异常过滤特征任一触发 → 快照 anomaly_flags 置位 → 决策 LLM 优先 NO_TRADE，并由代码兜底强制 NO_TRADE。

## 20. 风控设计

### 20.1 数据风控

```text
WS 断线、数据延迟超阈值、盘口为空、spread 异常、价格跳变异常 → anomaly_flags 置位 → 强制 NO_TRADE
```

### 20.2 LLM 判断风控

```text
LLM 输出 JSON 不合法 / 不符合 Schema → 不记录为正式信号，判 NO_TRADE
LLM 输出违反 HARD 规则 → 代码兜底强制改为 NO_TRADE，记录 invalid_conditions
LLM 理由与数据明显矛盾 / 置信度不足 → NO_TRADE
```

### 20.3 频率与稳定性风控

```text
连续错误 3 次 → 进入冷却期
短时间信号反复切换 → 暂停预测
市场剧烈波动 / 重大消息前后 → 降低频率或进入观察模式
```

### 20.4 LLM 调用风控（新增）

```text
决策 LLM 超时 / 报错 / 触发限频 → 本周期判 NO_TRADE，不伪造结果
设定单次调用超时上限与月成本预算上限，超限告警
```

## 22. 验收标准

### 22.1 数据验收

```text
行情数据连续性 >= 99%；WS 断线可自动重连；K线无明显缺口；盘口正常落库；
异常时正确置 anomaly_flags，绝无伪造数据当真实数据使用。
```

### 22.2 预测验收

```text
每分钟至少产出一次判断（含 NO_TRADE）
每条预测都有 p_up / p_down / confidence / target_time / 理由
每条预测记录 applied_rule_ids 与 rules_version
LLM 输出严格符合第 7.1 数据契约（字段、枚举、大小写）
每条预测 5 分钟后可自动验证
```

### 22.3 规则验收

```text
用户可增删改规则并即时生效于下一次预测
HARD 规则被违反时代码兜底强制 NO_TRADE
可按规则维度查看表现
```

### 22.4 复盘验收

```text
每条预测都有结果；每条错误预测都有 error_type；每天生成复盘报告；
LLM 能产出规则/提示词优化建议，且需人工确认才生效。
```

### 22.5 一致性验收（对应用户规则 7）

```text
LLM 输出 Schema = 数据库列 = API 响应，三者字段与枚举完全一致，无字段静默丢弃。
```

## 23. 推荐开发顺序

```text
第一步 数据采集：现货 K线/WS/订单簿/成交流 + 合约基础数据，落库并标记质量
第二步 特征系统：计算全部特征，产出市场快照（含 anomaly_flags）
第三步 规则系统：custom_rules 表 + 规则管理 API/界面 + 注入组装
第四步 决策 LLM：prompt 模板 + 结构化输出 + Schema 校验 + HARD 兜底 → 落库
第五步 复盘闭环：到期验证 + 复盘 LLM 归因 + 规则建议
第六步 报告与优化：日报、规则维度表现、提示词/规则版本管理与人工确认
第七步 增强（可选）：Vector DB / RAG 历史误判回灌；半自动交易扩展
```

## 24. 核心原则

```text
判断来自 LLM 的推理，而非机器学习概率
真实数据、真实接口；数据异常不伪造、不静默处理
用户规则可注入、可管理、可追踪
HARD 规则必须代码兜底，不只靠 LLM 自觉
LLM 只产出优化建议，线上变更由人工确认
NO_TRADE 是高质量判断的一部分
前后端、数据契约始终保持一致
```
